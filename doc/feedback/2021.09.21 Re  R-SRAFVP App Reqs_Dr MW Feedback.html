I am not sure if this will be the first time you have read through Software Specifications, and I would like to take a moment to describe how the two documents work together to create the foundation for the application work to come.

  • The Use Case Specification provides a high level visual and description for the major functions of the R-SRAFVP application.  This will not provide the details.  It will provide the general flow for each function for how it should work.  I recommend starting with the Intro and then using the diagram in conjunction with reading just the description for each use case.  Once you get the feel for what the application will do, then go for the use case details.  I hope it makes more sense that way.  
  • The Requirements Specification breaks down all of the functionality into explicit details.  The document is written to split requirements up by major areas of concern when creating an application (general, interface, functional and non-functional).  The requirements spec will cover details from the use cases and will include many more details needed to build the software.  Not all requirements will map back to use cases, just some.  Therefore, when you read this one, think about little things users may/may not do and see if your thoughts are included.
